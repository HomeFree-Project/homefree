# `/etc/nixos/homefree-configuration.nix` is generated, not authored

`/etc/nixos/homefree-configuration.nix` looks like an ordinary
hand-written NixOS module, but on every rebuild it is regenerated from
a string-literal template inside this repo. Editing only the deployed
file is a time bomb: it survives until the template happens to drift,
then is silently overwritten without warning.

## The pipeline

```
web-platform/backend/services/install.py
    └─ HOMEFREE_CONFIG_TEMPLATE = """{ config, lib, pkgs, ... }: ..."""
                       │
                       ▼
scripts/sync-template.py
    1. Read the template literal out of install.py.
    2. Read the deployed /etc/nixos/homefree-configuration.nix.
    3. Extract `users.users.<name>.hashedPassword` from the deployed
       file → bind the only two placeholders: `@@username@@`,
       `@@hashed_password@@`.
    4. Render: if rendered == deployed, emit `{"action":"noop"}`.
       If rendered != deployed, emit `{"action":"regenerate"}` and
       overwrite the deployed file (with a timestamped `.backup-*`).
                       │
                       ▼
scripts/sync-config.sh
    Called by BOTH:
      - scripts/build.sh (CLI rebuild path)
      - web-platform/backend/services/nix_operations._sync_config()
        (admin UI Apply path)
    So `./scripts/build.sh` and UI Apply go through the same sync —
    they are not separate code paths.
```

The deployed file has no `# AUTOGENERATED` header and no `DO NOT EDIT`
marker. Nothing about it signals that it's regenerated. A patch made
only there will appear to work until something else nudges the template.

## How the recent breakage happened

1. Added `enabled` to the mount schema in `module.nix` and the
   consumer in `modules/mounts.nix`.
2. Discovered the JSON→Nix loader's `mounts = map (m: { ... }) ...`
   block in `/etc/nixos/homefree-configuration.nix` was dropping the
   new field. Patched it there directly:
   `enabled = m.enabled or true;`
3. The next rebuild worked because the install.py template was already
   in sync with the rest of the deployed file → `sync-template` saw
   `rendered == existing` → noop → deployed-only patch survived.
4. A later commit (`Backups section UI refactor`) edited the template
   to add `extra-from-paths` normalization → next sync ran
   `rendered != existing` → regenerate → deployed file overwritten →
   the `enabled` patch was wiped → boot tried to mount three disabled
   mount-points → emergency mode.

## Rules

- **Any new JSON→Nix binding belongs in `HOMEFREE_CONFIG_TEMPLATE`
  inside `web-platform/backend/services/install.py`** — not only in
  `/etc/nixos/homefree-configuration.nix`. If you change the schema in
  `module.nix`, the corresponding loader update goes into the template.
- **Patching the deployed file is a valid hotfix only if you also
  patch the template the same turn.** Otherwise the deployed-file
  edit is a debt.
- **Verifying the fix took:** render the template with the deployed
  file's existing `@@username@@` and `@@hashed_password@@` substituted
  in, and confirm the result is byte-identical to the deployed file.
  When that holds, `sync-template.py` reports `noop` on the next
  rebuild and the patch is durable.
- **Only two placeholders in the template body** (`@@username@@` and
  `@@hashed_password@@`). The other 35 `@@...@@` tokens you'll find
  by grepping install.py belong to *other* generated artifacts
  (e.g. `homefree-encryption.nix`), not this template.
